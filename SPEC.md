# Rogue Core — Architecture Specification

```
version: 0.3.0-draft
status: brainstorm
date: 2026-03-08
```

## Overview

Rogue Core is a modular, open-source AI agent framework. It provides the plumbing for routing messages from any source (Telegram, CLI, Slack, etc.) through an IAM layer, into an AI agent orchestrator, and back. The framework is designed to be imported as a Go module by an instance repo that provides its own agents, powers, tools, and configuration.

```
github.com/kidkuddy/rogue-core    ← open source framework (this repo)
github.com/you/my-rogue-instance  ← your agents, powers, tools, config
```

### Component Names

| Component | Codename | Role |
|---|---|---|
| Message Bus | **Telepath** | Bidirectional message routing between sources and the pipeline |
| IAM | **Helmet** | Source validation, user/session management, power resolution |
| Agent Orchestrator | **Cerebro** | Spawns AI agents, manages providers, conversation state |
| Response Handler | **Warp** | Response routing, formatting, delivery |
| Storage | **Store** | Namespaced database and file storage |
| Scheduler | **Schedule** | Time-based task execution, cron support |
| Process Coordinator | **Coordinator** | Separate binary — process lifecycle, worker management, hot reload |

## 1. Telepath (Message Bus)

### Purpose

Bidirectional message bus that decouples message sources from the processing pipeline. Any source (Telegram bot, CLI, scheduler, another agent) emits messages in a unified format. Responses flow back through Telepath to the originating source (or a different one, for cross-channel replies).

### How It Was Done in v0.2

There was no message bus. The Telegram bot's `processMessage()` function was the entire pipeline — it received the Telegram update, ran IAM, called Claude, and sent the response back, all in one goroutine. Adding a new source meant rewriting the pipeline or duplicating it. The scheduler bypassed this by calling `HandleScheduledMessage()` directly on the bot, coupling it to Telegram. The outbox (for MCP tools sending messages) used a file-based queue polled every 2 seconds — a separate, parallel delivery mechanism with no shared abstraction.

### v0.3 Design

Telepath is a Go channel-based bus. Sources register as adapters that implement a `Source` interface. Each source:
- Emits inbound `Message` structs onto the bus
- Consumes outbound `Response` structs from the bus
- Handles its own transport-specific concerns (debouncing, file buffering, chunking)

```go
type Source interface {
    // ID returns a unique identifier for this source (e.g., "telegram:12345", "cli:main")
    ID() string

    // Start begins listening for inbound messages. Emit messages via the channel.
    Start(ctx context.Context, inbound chan<- Message) error

    // Send delivers an outbound response to this source's transport.
    Send(ctx context.Context, resp Response) error

    // Stop gracefully shuts down the source.
    Stop(ctx context.Context) error
}
```

```go
type Message struct {
    ID             string            // Unique message ID (UUID, generated by source)
    SourceID       string            // Source that emitted this ("telegram:12345", "scheduler")
    AgentID        string            // Target agent ("rogue", "doom")
    ChannelID      string            // Transport-level conversation ID (Telegram chat_id, Slack channel)
    UserID         string            // Sender identity (string to support non-numeric IDs)
    ChatType       string            // "private", "group", "scheduled", "agent"
    Text           string            // Message content
    Attachments    []Attachment      // Files, images, documents
    ReplyTo        *MessageRef       // If replying to a previous message
    Reply          bool              // Whether a response should be sent back (default: true)
    AgentTurnDepth int               // Incremented on agent-to-agent hops (system max enforced)
    Metadata       map[string]any    // Source-specific extras (Telegram message_id, Slack thread_ts)
}

type Attachment struct {
    Path     string
    Name     string
    MimeType string
}

type MessageRef struct {
    ID   string
    Text string
}

type Response struct {
    MessageID    string            // The inbound message this responds to
    TargetSource string            // Source to deliver to (usually same as inbound, can differ)
    ChannelID    string            // Channel to deliver to (usually same as inbound, can differ)
    Text         string            // Response content
    Attachments  []Attachment      // Files to send
    Metadata     map[string]any    // Source-specific delivery hints
}
```

### Sources to Implement

| Source | ID Format | Status |
|---|---|---|
| Telegram | `telegram:<bot_id>` | implement (port from v0.2) |
| CLI | `cli:<agent>` | stub |
| Scheduler | `scheduler` | implement (port from v0.2) |
| Agent (inter-agent) | `agent:<agent_id>` | stub |

### Notes

- Debouncing is source-specific. The Telegram source adapter handles its own 5s debounce buffer before emitting to Telepath.
- File attachment buffering (Telegram sends files and text as separate updates) is handled inside the Telegram source adapter. By the time a Message hits Telepath, attachments are resolved.
- Cross-channel responses: Warp can emit a Response with a different `TargetSource`/`ChannelID` than the inbound message. Telepath routes it to the correct source adapter.

## 2. Helmet (IAM)

### Purpose

Identity and access management layer. Validates sources, manages users and sessions, resolves the agent's persona and powers for a given message context. Sits between Telepath (inbound) and Cerebro (orchestration).

### How It Was Done in v0.2

IAM was a 10-step procedural function (`iam.Process()` in `internal/iam/process.go`) that mutated a `ChatContext` struct. It:
1. Created/fetched a user record from `iam.sqlite`
2. Created/fetched a session keyed on `(bot_id, chat_id)`
3. Stored the incoming message in a `messages` table
4. Checked ownership via `ctx.UserID == cfg.OwnerID`
5. Queried `user_power_sets` for bundles scoped to `(bot_id, user_id, chat_id)`
6. Loaded each power set file from disk 3 times (tools, dirs, instructions — separately)
7. Built Claude CLI arguments (`--allowedTools`, `--add-dir`, `--system-prompt`)

Problems: tightly coupled to Telegram (used `tgbotapi.Message` to create context), opened separate DB connections per query, loaded power set files 3x per message, and hardcoded the "owner" concept to a single env var comparison.

### v0.3 Design

Helmet receives a `Message` from Telepath and produces an `EnrichedMessage` for Cerebro. It's a pure function over Store data — no side effects beyond user/session creation.

```go
type Helmet interface {
    // Process validates the message and enriches it with session/power/persona data.
    // Returns an error if the source or user is rejected.
    Process(ctx context.Context, msg Message) (*EnrichedMessage, error)
}

type EnrichedMessage struct {
    Message                          // Original message (embedded)
    ChatID        string             // Agentic conversation ID (UUID), mapped from channel
    SessionID     string             // Session identifier
    User          User               // Resolved user record
    Agent         AgentConfig        // Resolved agent persona
    PowerSet      PowerSet           // Resolved union of all assigned powers
    Tags          []string           // Session/user tags (admin-configured + auto-generated)
    IsRoot        bool               // Whether this user is root (resolved via RootResolver)
}

type User struct {
    ID        string
    Username  string
    FirstName string
    CreatedAt time.Time
    LastSeen  time.Time
}

type AgentConfig struct {
    ID       string  // "rogue", "doom"
    Persona  string  // Contents of agent markdown file
}

type PowerSet struct {
    Powers       []Power            // Individual powers in this set
    Tools        []string           // Union of all tools from all powers
    Directories  []string           // Union of all directories from all powers
    Instructions string             // Combined instructions from all powers
}

type Power struct {
    Name         string             // "memory", "queue", "scheduler"
    Tools        []string           // MCP tool names this power grants
    Directories  []string           // Filesystem paths this power grants
    Instructions string             // Usage instructions for the agent
}
```

### Root Resolver

v0.2 hardcoded `ctx.UserID == cfg.OwnerID`. v0.3 uses a configurable lambda:

```go
type RootResolver func(userID string) bool
```

Configured in YAML:

```yaml
helmet:
  root_resolver:
    type: env_match
    env: OWNER_ID
```

Could support other types later: `webhook`, `group_membership`, `always_false`, etc.

### Session Mapping

```
(SourceID, ChannelID, AgentID) → ChatID (UUID)
```

- A user DMing Agent Rogue via Telegram bot A: `("telegram:botA", "123456", "rogue")` → `chat-uuid-1`
- Same user, same agent, via Slack: `("slack:workspace", "C789", "rogue")` → `chat-uuid-2` (different session)
- Same Telegram chat, different agent: `("telegram:botB", "123456", "doom")` → `chat-uuid-3`

Helmet creates the ChatID on first encounter and stores the mapping in `Store("iam")`.

### Power Resolution

1. Query assigned powers for `(agent_id, user_id, channel_id)` from Store
2. Load each power definition from disk (once, cached per process lifetime or with file watcher)
3. Build the union: `PowerSet = ∪ power.Tools, ∪ power.Dirs, concat power.Instructions`
4. Return as a flat `PowerSet` struct

### Auto-Generated Tags

Every message gets default tags appended:

```
user:<user_id>
source:<source_id>
channel:<channel_id>
agent:<agent_id>
```

Admin-configured tags (from Store) are merged. Used by Cerebro for provider routing and by Store for cost aggregation.

## 3. Cerebro (Agent Orchestrator)

### Purpose

Spawns and manages AI agent sessions. Receives enriched messages from Helmet, selects the right provider, builds the agent configuration (persona, tools, conversation history), executes the agent, and emits results.

### How It Was Done in v0.2

`ai.RunClaudeCode()` in `internal/ai/claude_code.go` shelled out to `claude --print --output-format json`. The IAM layer built the CLI arguments directly (`--allowedTools`, `--system-prompt`, `--resume`). Claude Code managed its own MCP server lifecycle — spawning tool binaries as child processes based on `.mcp.json`. Session resumption was via `--resume <session_id>` where the session ID was stored in `iam.sqlite`.

The function was ~60 lines, tightly coupled to Claude Code's CLI interface, and returned a parsed JSON response with content, usage stats, and session ID.

### v0.3 Design

Cerebro is provider-agnostic. It defines an `AgentProvider` interface that Claude Code implements, with room for others.

```go
type Cerebro interface {
    // Execute runs an agent session and returns the result.
    Execute(ctx context.Context, msg *EnrichedMessage) (*AgentResult, error)
}

type AgentProvider interface {
    // ID returns the provider identifier ("claude-code", "copilot", etc.)
    ID() string

    // Execute runs the agent with the given configuration.
    Execute(ctx context.Context, req AgentRequest) (*AgentResult, error)
}

type AgentRequest struct {
    ChatID          string            // Agentic conversation ID
    SessionState    any               // Provider-specific session data (claude session ID, openai thread ID, etc.)
    Persona         string            // System prompt / persona
    Prompt          string            // User message
    Attachments     []Attachment      // Files
    Tools           []string          // Allowed MCP tool names
    Directories     []string          // Allowed filesystem paths
    Instructions    string            // Power instructions (appended to system prompt)
    MCPConfigPath   string            // Path to dynamically generated MCP config
    Tags            []string          // Session tags (for provider selection, model routing)
    MaxTurns        int               // Safety limit
    Env             map[string]string // Environment variables for the agent process
}

type AgentResult struct {
    Content      string             // Agent's response text
    SessionState any                // Updated provider-specific session data (to persist)
    Usage        Usage              // Token/cost metrics
    HitMaxTurns  bool               // Whether the agent hit the turn limit
    Metadata     map[string]any     // Provider-specific extras
}

type Usage struct {
    InputTokens      int64
    OutputTokens     int64
    CacheReadTokens  int64
    CacheWriteTokens int64
    CostUSD          float64
    DurationMS       int64
    NumTurns         int
}
```

### Dynamic MCP Config Generation

v0.2 used a static `.mcp.json` file. v0.3 generates it per-session based on the resolved PowerSet:

1. Helmet resolves the PowerSet (list of tools)
2. Cerebro maps tool names to MCP server configurations (from a registry)
3. Cerebro writes a temporary `.mcp.json` scoped to this session
4. The agent provider receives the path and uses it

```go
type MCPRegistry interface {
    // GenerateConfig builds an MCP config file containing only the servers
    // needed for the given tool list. Returns the path to the temp file.
    GenerateConfig(tools []string, env map[string]string) (string, error)
}
```

This means an agent only sees MCP servers for tools in its PowerSet. No extra servers running.

### Conversation History

Cerebro owns conversation state. It saves/loads from `Store("conversations")`:

- Before execution: load `SessionState` for this `ChatID` from Store
- After execution: save updated `SessionState` to Store

The `SessionState` is `any` because it's provider-specific:
- Claude Code: `string` (session ID for `--resume`)
- OpenAI: `string` (thread ID)
- Local models: `[]Message` (full history)

### Agent-to-Agent Communication

When an agent wants to message another agent, it emits a message through Telepath with:
- `SourceID: "agent:<sender_agent_id>"`
- `AgentID: "<target_agent_id>"`
- `AgentTurnDepth: current_depth + 1`

Cerebro checks `AgentTurnDepth` before executing. If it exceeds the system max (configurable, default 3), the message is rejected and the original user is tagged for manual handling.

Instructions injected for agent-to-agent contexts:
```
You are responding to another agent. Be pragmatic — aim to resolve in a single response.
If this requires human judgment, tag the user instead of continuing the agent chain.
```

## 4. Warp (Response Handler)

### Purpose

Takes agent results and routes them to the correct destination. Handles response suppression, cross-channel routing, queue redirection, and format adaptation per source.

### How It Was Done in v0.2

Response handling was embedded in `bot.processMessage()`. After Claude returned, the bot:
1. Sent a typing indicator (refreshed every 4s)
2. Recorded usage stats to `usage_stats` table
3. Updated the session's `claude_session_id`
4. Split the response at 4096-char boundaries (paragraph → line → hard cut)
5. Sent chunks via Telegram API
6. Added emoji reactions based on turn count (⚡ for 3+, 🧠 for 10+)
7. Cleared session messages

For MCP tool responses (rogue-telegram), a separate file-based outbox was polled every 2 seconds. Two completely separate delivery paths.

### v0.3 Design

Warp receives `AgentResult` + the original `EnrichedMessage` context. It decides what to do.

```go
type Warp interface {
    // Handle processes an agent result and routes the response.
    Handle(ctx context.Context, msg *EnrichedMessage, result *AgentResult) error
}
```

### Routing Logic

1. **Check `msg.Reply` flag**: if `false`, suppress response to source. May still log or push to queue.
2. **Check for redirect**: if `msg.Metadata["response_target"]` specifies a queue, push result there instead of replying.
3. **Record usage**: write to `Store("iam")` — tokens, cost, duration, tags, session.
4. **Save conversation state**: tell Cerebro to persist the updated `SessionState`.
5. **Build Response**: construct `Response` struct with `TargetSource`, `ChannelID`, content.
6. **Emit to Telepath**: Telepath routes to the correct source adapter, which handles transport-specific formatting (Telegram chunking, Slack threading, etc.).

### Cross-Channel Responses

An agent can instruct Warp to reply to a different channel:

```go
// In AgentResult.Metadata:
{
    "cross_channel": {
        "target_source": "telegram:98765",
        "target_channel": "group-chat-id"
    }
}
```

Warp builds a Response targeting that source/channel. Telepath routes it.

## 5. Store (Namespaced Storage)

### Purpose

Centralized storage layer providing namespaced SQLite databases and file storage. Every component and every power gets its own isolated namespace. Store handles security boundaries, migrations, and backups.

### How It Was Done in v0.2

Storage was scattered:
- `internal/memory/db.go` opened SQLite databases by name from `$ROGUE_DATA/dbs/`
- Each MCP tool server opened its own DB connections independently
- The scheduler opened `queues.sqlite` directly to push results (bypassing the queue MCP server)
- File operations in `tools/memory/` read/wrote to `$ROGUE_DATA/files/` with no path validation
- Backup was a git commit triggered by the `backup` MCP tool
- 6+ separate SQLite files with no coordinated access

### v0.3 Design

Store is a Go library for core components and an MCP server for agent-facing access.

```go
type Store interface {
    // Namespace returns an isolated storage scope.
    Namespace(name string) Namespace

    // Backup checkpoints all databases and runs backup procedure.
    Backup(ctx context.Context) error

    // Close cleanly shuts down all open databases.
    Close() error
}

type Namespace interface {
    // DB returns a SQLite database handle for this namespace.
    // The database file lives at <data_dir>/db/<namespace>/store.sqlite
    DB() (*sql.DB, error)

    // FilePath returns the absolute path for a named file in this namespace.
    // Files live at <data_dir>/files/<namespace>/
    FilePath(name string) string

    // FileList returns all files in this namespace.
    FileList() ([]FileInfo, error)

    // ReadFile reads a file from this namespace.
    ReadFile(name string) ([]byte, error)

    // WriteFile writes a file to this namespace.
    WriteFile(name string, data []byte) error
}
```

### Directory Structure

```
<data_dir>/
├── db/
│   ├── iam/
│   │   └── store.sqlite          ← Helmet's database
│   ├── conversations/
│   │   └── store.sqlite          ← Cerebro's conversation history
│   ├── scheduler/
│   │   └── store.sqlite          ← Scheduler's tasks
│   ├── queue/
│   │   └── store.sqlite          ← Queue data
│   ├── power:memory/
│   │   └── store.sqlite          ← Agent-facing memory power
│   ├── power:phd/
│   │   └── store.sqlite          ← Agent-facing PhD research
│   ├── power:contacts/
│   │   └── store.sqlite          ← Agent-facing contacts
│   └── power:running/
│       └── store.sqlite
├── files/
│   ├── iam/
│   ├── conversations/
│   ├── power:memory/
│   │   ├── jobs/
│   │   ├── resume/
│   │   └── scratchpad/
│   └── power:phd/
│       └── papers/
└── config.yaml
```

### Access Control

- Core components (Helmet, Cerebro, Warp) access their namespaces directly via the Go library.
- Agent-facing access goes through an MCP server (`rogue-store`) that receives the namespace as a parameter.
- The MCP server validates that the requesting agent's PowerSet includes the requested namespace.
- A power `memory` maps to namespace `power:memory`. A power cannot access `iam` or `conversations`.

### Standard Namespaces

| Namespace | Owner | Purpose |
|---|---|---|
| `iam` | Helmet | Users, sessions, powers, usage stats |
| `conversations` | Cerebro | Provider-specific session state |
| `scheduler` | Scheduler | Scheduled tasks |
| `queue` | Queue system | Named queues and items |
| `power:<name>` | Agent (via power) | Power-scoped data |

### Migration Pattern

Same as v0.2: `ALTER TABLE ... ADD COLUMN` that silently fails if column exists. Never destructive DDL. Each namespace owns its own schema.

## 6. Schedule

### Purpose

Time-based task execution. Emits messages to Telepath when tasks are due. Also exposes an MCP tool interface for agents to create/cancel/list tasks.

### How It Was Done in v0.2

The scheduler was a goroutine inside `cmd/rogue/main.go` that ticked every 30 seconds. It queried `scheduled_tasks` from `threads.sqlite`, marked tasks as `running`, called `HandleScheduledMessage()` on the correct bot (looked up by `bot_id` from a handler map), then marked tasks `done` or `failed`. Cron tasks were rescheduled automatically. On startup, it recovered tasks stuck in `running` state.

The scheduler opened `queues.sqlite` directly to push results to named queues — bypassing the queue MCP server entirely. This was noted as a pragmatic but architecturally inconsistent choice.

### v0.3 Design

The scheduler is a **source on Telepath**. When a task fires, it emits a `Message` with:
- `SourceID: "scheduler"`
- `AgentID`: from task record
- `ChannelID`: from task record (for response routing)
- `Reply`: from task record (`require_reply`)
- `ChatType: "scheduled"`

```go
type Scheduler interface {
    // Start begins the tick loop.
    Start(ctx context.Context, bus chan<- Message) error

    // Stop gracefully drains in-flight tasks.
    Stop(ctx context.Context) error

    // Schedule creates a new task.
    Schedule(task ScheduledTask) (string, error)

    // Cancel cancels a pending task.
    Cancel(taskID string) error

    // List returns tasks filtered by status.
    List(status string) ([]ScheduledTask, error)

    // Delay postpones a pending task.
    Delay(taskID string, duration time.Duration) error
}

type ScheduledTask struct {
    ID           string
    AgentID      string            // Which agent should handle this
    UserID       string            // Who created it
    ChannelID    string            // Where to route the response
    SourceID     string            // Original source for response routing
    MessageText  string            // The prompt to send
    ScheduledFor time.Time
    CronExpr     string            // Optional: recurring schedule
    Reply        bool              // Whether to send response back
    Queue        string            // Optional: push result to this queue
    Status       string            // pending | running | done | failed | cancelled
    Tags         []string          // Inherited from creating session
}
```

### Queue Bridge

When a task has a `Queue` field, after execution Warp pushes the result to the named queue. This is Warp's responsibility now (not the scheduler's), since Warp handles all response routing.

## 7. Coordinator (Process Lifecycle)

### Purpose

Main process that manages the lifecycle of all workers. Handles hot reload, binary updates, and graceful shutdown.

### How It Was Done in v0.2

The main process did everything: bot polling, IAM, Claude invocation, scheduling. Restart was `syscall.Exec()` (replace the process in-place) triggered by SIGHUP or a `.restart` file. This had problems: child processes (Claude sessions, MCP servers) weren't always cleaned up, sometimes spawning zombie processes. No graceful drain — in-flight Claude calls were killed on restart.

### v0.3 Design

The coordinator is a **separate binary** from the pipeline. This is critical: if rogue-core is rebuilt mid-conversation, the coordinator (which barely imports anything from core) stays alive and restarts the pipeline worker with the new binary. The coordinator is intentionally thin — it's a process supervisor scoped to Rogue.

```
Coordinator (separate binary, rarely rebuilt)
  └── Pipeline Worker (separate binary, rebuilt when core changes)
        ├── Telepath + Helmet + Cerebro + Warp (goroutines)
        ├── Source adapters (goroutines)
        ├── Schedule tick loop (goroutine)
        └── MCP Tool Servers (child processes, spawned by agent provider)
```

The **pipeline binary** is what the instance repo compiles. It imports rogue-core and runs the full pipeline. The coordinator just manages its lifecycle.

```go
// Coordinator — thin supervisor, separate binary
type Coordinator struct {
    pipelineBin string            // Path to pipeline binary
    pipeline    *os.Process       // Running pipeline process
    configPath  string            // Watched for changes
}
```

```go
// Pipeline — the actual work, compiled by instance repo
type Pipeline struct {
    config   *Config
    store    Store
    telepath *Telepath
    helmet   Helmet
    cerebro  Cerebro
    warp     Warp
    schedule Schedule
}
```

### Hot Reload

- **Config change** (YAML modified): Coordinator detects via file watcher. Sends SIGUSR1 to pipeline. Pipeline reloads config in-process (Viper WatchConfig). No restart needed.
- **MCP tool binary rebuilt**: Agent provider spawns fresh MCP servers per session. Next agent session picks up the new binary automatically. No restart needed.
- **Pipeline binary rebuilt** (rogue-core or instance code changed): Coordinator detects binary change (mtime or checksum). Sends SIGTERM to pipeline. Pipeline drains in-flight messages (up to 30s), exits. Coordinator spawns new pipeline process.
- **Coordinator binary rebuilt**: Rare. External supervisor (systemd, launchd) restarts it. Coordinator exits cleanly, pipeline also exits (child process).

### Communication

- Coordinator ↔ Pipeline: Unix signals (SIGTERM for shutdown, SIGUSR1 for config reload) + stderr for health/logs.
- Within Pipeline: Go channels between goroutines. All components run in the same process.

```go
func (c *Coordinator) Start() error {
    // 1. Start pipeline binary as child process
    // 2. Watch pipeline binary for changes (mtime/checksum)
    // 3. Watch config file for changes
    // 4. Handle signals (SIGTERM → graceful shutdown of pipeline)
    // 5. Restart pipeline on binary change or crash
}
```

## 8. Configuration

### How It Was Done in v0.2

Configuration was loaded from `.env` via `godotenv`. Values: `TELEGRAM_BOT_TOKENS` (comma-separated), `OWNER_ID`, `OWNER_NAME`, `ROGUE_DATA`. Agent-to-bot mapping was via `AGENT_BOT_<bot_id>` env vars. MCP tool configuration lived in `.mcp.json`. Power sets were files on disk. No structured config file — everything was env vars or files.

### v0.3 Design

Viper YAML as the primary config, with `.env` support. YAML values can reference env vars via `env:VAR_NAME` syntax, resolved at load time.

```yaml
# config.yaml

store:
  data_dir: "env:ROGUE_DATA"

helmet:
  root_resolver:
    type: env_match
    env: OWNER_ID
  powers_dir: ./powers        # Power definition files
  agents_dir: ./agents        # Agent persona files

cerebro:
  default_provider: claude-code
  max_turns: 100
  max_agent_depth: 3          # Agent-to-agent hop limit
  providers:
    claude-code:
      binary: claude
      # MCP tool registry — maps tool names to server configs
      tools:
        rogue-store:
          command: "env:ROGUE_BIN/rogue-store"
          env:
            ROGUE_DATA: "env:ROGUE_DATA"
        rogue-scheduler:
          command: "env:ROGUE_BIN/rogue-scheduler"
          env:
            ROGUE_DATA: "env:ROGUE_DATA"
        # Instance-specific tools added here

telepath:
  sources:
    - type: telegram
      id: "telegram:rogue"
      token: "env:TELEGRAM_BOT_TOKEN_ROGUE"
      agent: rogue
      debounce_ms: 5000
    - type: telegram
      id: "telegram:doom"
      token: "env:TELEGRAM_BOT_TOKEN_DOOM"
      agent: doom
      debounce_ms: 5000
    - type: cli
      id: "cli:main"
      agent: rogue

scheduler:
  tick_interval: 30s
  task_timeout: 10m
  default_queue: review
```

### Env Substitution

At load time, any YAML value prefixed with `env:` is replaced with the corresponding environment variable:

```go
func resolveEnv(val string) string {
    if strings.HasPrefix(val, "env:") {
        return os.Getenv(strings.TrimPrefix(val, "env:"))
    }
    return val
}
```

## 9. Powers (Capability Definitions)

### How It Was Done in v0.2

Called "power sets" (the bundle itself). Each was a markdown file with YAML frontmatter in `power_sets/`. Frontmatter defined `tools` (list of MCP tool names), optional `directories` (filesystem paths). The markdown body was instructions injected into the agent's system prompt. IAM loaded these from disk up to 3 times per message (once for tools, once for dirs, once for instructions).

### v0.3 Design

A **Power** is a named capability bundle. A user's **PowerSet** is the resolved union of their assigned powers. Power files live in `powers/` (not `power_sets/`).

```yaml
# powers/memory.md
---
name: memory
description: Raw data access — SQL queries, file operations, backups
namespace: power:memory
tools:
  - rogue-store__sql
  - rogue-store__file_read
  - rogue-store__file_write
  - rogue-store__file_list
  - rogue-store__db_list
  - rogue-store__backup
directories:
  - "env:ROGUE_DATA/files"
---

## Instructions

You have access to the memory store. Use SQL for structured data and files for documents.

Available databases in namespace `power:memory`:
- rogue: general purpose
- finances: financial tracking
...
```

Key changes from v0.2:
- `namespace` field ties the power to a Store namespace (access control)
- Tools reference the centralized `rogue-store` MCP server with namespace scoping
- Power files are loaded once at startup and cached (file watcher for changes)
- The `env:` substitution works in power files too

## 10. Full Pipeline Flow

```
┌─────────────┐     ┌──────────┐     ┌────────┐     ┌─────────┐     ┌──────┐
│   Sources   │────▶│ Telepath  │────▶│ Helmet │────▶│ Cerebro │────▶│ Warp │
│ (Telegram,  │     │  (Bus)   │     │ (IAM)  │     │ (Agent) │     │(Resp)│
│  CLI, Sched)│◀────│          │◀────│        │     │         │────▶│      │
└─────────────┘     └──────────┘     └────────┘     └─────────┘     └──────┘
                                          │               │              │
                                          ▼               ▼              ▼
                                     ┌─────────────────────────────────────┐
                                     │         Store (Namespaced)          │
                                     │  iam | conversations | scheduler   │
                                     │  queue | power:* namespaces        │
                                     └─────────────────────────────────────┘
```

### Request Lifecycle

1. **Source** receives external message (Telegram update, CLI input, scheduler tick)
2. **Source** normalizes to `Message` struct, handles transport concerns (debounce, file buffering)
3. **Telepath** routes `Message` to pipeline
4. **Helmet** validates source, resolves user (create if new), maps `(source, channel, agent)` → `chat_id`
5. **Helmet** checks root status via `RootResolver`
6. **Helmet** loads assigned powers from `Store("iam")`, builds `PowerSet` union, loads agent persona
7. **Helmet** generates auto-tags, merges admin-configured tags
8. **Helmet** emits `EnrichedMessage` to Cerebro
9. **Cerebro** loads conversation state from `Store("conversations")`
10. **Cerebro** generates dynamic MCP config based on PowerSet tools
11. **Cerebro** selects provider based on tags
12. **Cerebro** builds `AgentRequest`, spawns agent via provider
13. **Agent** executes (Claude Code CLI, etc.), calls MCP tools as needed
14. **Cerebro** captures `AgentResult`, saves updated conversation state
15. **Warp** receives result + original enriched message
16. **Warp** checks `Reply` flag — suppress if false
17. **Warp** records usage to `Store("iam")` with tags
18. **Warp** builds `Response`, emits to Telepath
19. **Telepath** routes `Response` to correct source adapter
20. **Source** delivers response via transport (Telegram API, stdout, etc.)

## 11. Open Source Boundary

### rogue-core (this repo) provides:

- All interfaces (`Source`, `Store`, `Helmet`, `Cerebro`, `Warp`, `AgentProvider`, `Scheduler`)
- Built-in implementations:
  - `TelegramSource` — Telegram bot adapter
  - `CLISource` — CLI adapter (stub)
  - `SQLiteStore` — Namespaced SQLite + file storage
  - `DefaultHelmet` — IAM implementation
  - `DefaultCerebro` — Agent orchestrator
  - `DefaultWarp` — Response handler
  - `ClaudeCodeProvider` — Claude Code agent provider
  - `DefaultScheduler` — Cron + one-shot scheduler
  - `Coordinator` — Process lifecycle manager
- Standard MCP tools:
  - `rogue-store` — Namespaced SQL + file access
  - `rogue-scheduler` — Task management
  - `rogue-iam` — Power/user management (for agents to query permissions)
- Config loading (Viper + env substitution)
- Power file parser

### Instance repo (private) provides:

- `main.go` — imports rogue-core, registers custom tools, starts coordinator
- `config.yaml` — instance-specific configuration
- `.env` — secrets
- `agents/` — persona files (rogue.md, doom.md, etc.)
- `powers/` — power definitions (memory.md, phd.md, etc.)
- `tools/` — custom MCP tool servers (phd, scraper, contacts, etc.)
- `Makefile` — build targets for custom tools

### Instance main.go Example

```go
package main

import (
    "log"
    "github.com/kidkuddy/rogue-core/core"
)

func main() {
    coord, err := core.NewCoordinator("config.yaml")
    if err != nil {
        log.Fatal(err)
    }

    // Register instance-specific MCP tools (added to tool registry)
    coord.RegisterTool("rogue-phd", core.MCPTool{
        Command: "./bin/rogue-phd",
        Env:     map[string]string{"ROGUE_DATA": "${ROGUE_DATA}"},
    })
    coord.RegisterTool("rogue-scraper", core.MCPTool{
        Command: "./bin/rogue-scraper",
    })

    if err := coord.Start(); err != nil {
        log.Fatal(err)
    }
}
```
